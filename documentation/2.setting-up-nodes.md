This document details the complete setup of the Proxmox VE cluster that underpins the homelab
infrastructure. Follow the steps in order; each section depends on the one before it.

## 1. Hardware Overview

The cluster is built from two repurposed consumer machines. The intention is to add further nodes
as more hardware becomes available.

| Node | Form Factor | Role |
|------|-------------|------|
| `pve-node1` | Mini PC | Primary node — hosts the cluster quorum |
| `pve-node2` | Old laptop | Secondary node — additional compute and storage |
| *(future)* | TBD | Additional nodes to expand capacity |

> **Quorum note:** A two-node Proxmox cluster has no automatic quorum. Both nodes must remain
> reachable for the cluster to operate without manual intervention. When a third node is added
> the cluster will gain true quorum and automatic failover.

### Minimum Requirements per Node

| Resource | Minimum |
|----------|---------|
| CPU | 64-bit, 2 cores |
| RAM | 4 GB (8 GB+ recommended) |
| Storage (OS) | 32 GB |
| Network | 1 Gbps Ethernet (Wi-Fi is not supported for cluster traffic) |

> Both machines must be connected via wired Ethernet before proceeding. Wireless interfaces are
> not supported for Proxmox cluster communication.

## 2. Installing Proxmox VE

Perform the steps in this section **on each node independently** before moving on to cluster
formation.

### 2.1 Download the Proxmox VE ISO

Download the latest Proxmox VE ISO from the official site:
[https://www.proxmox.com/en/downloads](https://www.proxmox.com/en/downloads)

Flash the ISO to a USB drive (8 GB or larger). On macOS or Linux, use a tool such as
[Balena Etcher](https://etcher.balena.io/) or `dd`.

### 2.2 Boot the Installer

1. Insert the USB drive into the target machine.
2. Boot from the USB. Enter the BIOS/UEFI boot menu (typically `F2`, `F12`, or `DEL` at POST)
   and select the USB drive.
3. At the Proxmox VE boot menu, select **Install Proxmox VE (Graphical)**.

### 2.3 Follow the Graphical Installer

| Installer Step | Recommended Value |
|----------------|-------------------|
| Target disk | Select the internal drive. Enable ZFS (RAID0) for single-disk installs. |
| Country / Timezone / Keyboard | Set to your locale. |
| Root password | Choose a strong, unique password. Record it securely. |
| Hostname (FQDN) | `pve-node1.local` / `pve-node2.local` |
| IP address | Static IP on your LAN (e.g. `192.168.0.105` / `192.168.0.106`) |
| Netmask | `255.255.255.0` |
| Gateway | Your router's LAN IP (e.g. `192.168.0.1`) |
| DNS server | Your LAN DNS or `1.1.1.1` |

Click **Install** and allow the process to complete. The machine will reboot automatically.

### 2.4 Post-Install Verification

After reboot, log in via the web UI at `https://<node-ip>:8006` (accept the self-signed TLS
warning) using:

- **Username:** `root`
- **Realm:** `Linux PAM standard authentication`
- **Password:** the root password set during installation

Confirm the node is visible in the web UI and that disk and network resources are listed
correctly under the node's **Summary** tab.

### 2.5 Disable the Enterprise Repository (No Subscription)

By default Proxmox VE is configured to use the enterprise apt repository, which requires a paid
subscription. On each node, switch to the no-subscription repository:

1. In the web UI, navigate to **Node → Updates → Repositories**.
2. Disable the entry labelled `pve-enterprise`.
3. Add the community repository:
   - Click **Add** and select `No-Subscription` from the drop-down.
4. Open a root shell on the node (via the web UI **Shell** or SSH) and run:

```bash
apt-get update && apt-get upgrade -y
```

Repeat this entire section on the second node before proceeding.

## 3. Forming the Proxmox Cluster

A Proxmox VE cluster allows both nodes to be managed from a single web UI and enables VM
migration between nodes. Cluster formation must be completed **before** any VMs or containers are
created; adding a node with existing guests to an existing cluster is not supported.

### 3.1 Create the Cluster on the Primary Node (`pve-node1`)

1. Log in to the web UI of `pve-node1`.
2. Navigate to **Datacenter → Cluster → Create Cluster**.
3. Enter a cluster name (e.g. `homelab-cluster`).
4. Leave **Cluster Network** set to the primary NIC's IP. Click **Create**.

Once the cluster is created, retrieve the join token:

1. Navigate to **Datacenter → Cluster → Join Information**.
2. Click **Copy Information**. Keep this page open or save the token — it is needed in the next
   step.

### 3.2 Join the Secondary Node (`pve-node-02`) to the Cluster

1. Log in to the web UI of `pve-node2`.
2. Navigate to **Datacenter → Cluster → Join Cluster**.
3. Paste the join information copied from `pve-node-01`.
4. Enter the `root` password of **`pve-node-01`** when prompted (not the local root password).
5. Click **Join**.

The join process takes approximately 30–60 seconds. When complete, `pve-node-02` disappears from
its own standalone UI and both nodes become visible when logging in to either node's IP.

### 3.3 Verify the Cluster

From either node's web UI, navigate to **Datacenter → Cluster**. Both nodes should be listed
with a green status indicator. Verify via the shell as well:

```bash
pvecm status
```

Expected output includes `Quorate: Yes` and both nodes listed as `online`.

## 4. Upgrading Proxmox VE and Removing the License Nag

The [ProxMenux](https://github.com/MacRimi/ProxMenux) utility provides a menu-driven interface
for common Proxmox VE post-install tasks, including in-place version upgrades and removal of the
subscription nag dialog. Run it on **each node separately**.

### 4.1 Run ProxMenux

Open a root shell on the node (web UI **Shell** or SSH as `root`) and execute:

```bash
bash <(curl -s https://raw.githubusercontent.com/MacRimi/ProxMenux/main/proxmenux.sh)
```

> **Security note:** Always review the contents of a remote script before piping it to bash.
> The ProxMenux source is publicly available at the link above.

### 4.2 Upgrade Proxmox VE to the Latest Version

1. In the ProxMenux main menu, select **Post Install**.
2. Select **Upgrade Proxmox VE**.
3. Confirm when prompted. The script updates apt sources, performs the upgrade, and reports any
   held-back packages.
4. Reboot the node after the upgrade completes:

```bash
reboot
```

After the reboot, confirm the new version:

```bash
pveversion
```

### 4.3 Remove the License Nag Dialog

The subscription-required pop-up in the web UI can be suppressed without a paid license:

1. Re-run ProxMenux on the node (same command as above).
2. Select **Post Install → Remove Subscription Nag**.
3. Confirm. The script patches the relevant JavaScript file in the web UI.
4. Hard-refresh the browser (`Ctrl+Shift+R` / `Cmd+Shift+R`) to clear the cached page.

Repeat steps 4.1–4.3 on the second node.

## 5. User Management

The default `root` account created during Proxmox VE installation holds unrestricted superuser
access at both the Proxmox API level and the underlying Linux OS. Two additional accounts are
created to reduce reliance on the root account and to separate operational concerns.

### 5.1 Administrative User (Personal Account)

This account provides day-to-day access to the Proxmox web UI and API with full administrative
rights over the virtualisation layer, but it operates **without Linux OS-level root access**.

#### Create the User

1. In the web UI, navigate to **Datacenter → Permissions → Users**.
2. Click **Add**.

| Field | Value |
|-------|-------|
| User name | `<your-username>` |
| Realm | `Proxmox VE authentication server` (pve) |
| Password | A strong, unique password |
| First name / Last name | Your name |
| Enabled | Yes |

3. Click **Add** to save.

#### Assign the Administrator Role

1. Navigate to **Datacenter → Permissions**.
2. Click **Add → User Permission**.

| Field | Value |
|-------|-------|
| Path | `/` (root path — grants permissions across all resources) |
| User | `<your-username>@pve` |
| Role | `Administrator` |
| Propagate | Checked |

3. Click **Add**.

> The built-in `Administrator` role includes all Proxmox API privileges. It does **not** grant
> SSH login or Linux OS-level root access, making it less privileged than the `root@pam` account
> which bypasses all Proxmox permission checks.

Use this account for all routine web UI and API interactions. Reserve `root@pam` for
break-glass scenarios only.

### 5.2 Infrastructure-as-Code (IaC) User

This account is used exclusively by OpenTofu (or Terraform) to provision and manage resources
on the cluster programmatically. It is granted only the permissions required for that purpose,
following the principle of least privilege.

#### Create the User

1. Navigate to **Datacenter → Permissions → Users → Add**.

| Field | Value |
|-------|-------|
| User name | `iac-provisioner` |
| Realm | `Proxmox VE authentication server` (pve) |
| Password | A strong, randomly generated password |
| Comment | `OpenTofu/Terraform automation account` |
| Enabled | Yes |

2. Click **Add**.

#### Create a Custom Role for IaC

Rather than assigning a broad built-in role, create a scoped role that grants only the privileges
required for VM and container provisioning:

1. Navigate to **Datacenter → Permissions → Roles → Create**.
2. Enter role name: `IaCProvisioner`.
3. Add the following privileges:

| Privilege Group | Privileges |
|-----------------|------------|
| Datastore | `Datastore.AllocateSpace`, `Datastore.Audit` |
| SDN | `SDN.Use` |
| VM | `VM.Allocate`, `VM.Audit`, `VM.Clone`, `VM.Config.CDROM`, `VM.Config.CPU`, `VM.Config.Cloudinit`, `VM.Config.Disk`, `VM.Config.HWType`, `VM.Config.Memory`, `VM.Config.Network`, `VM.Config.Options`, `VM.Console`, `VM.Migrate`, `VM.Monitor`, `VM.PowerMgmt` |
| Pool | `Pool.Audit` |

4. Click **Create**.

#### Assign the Role

1. Navigate to **Datacenter → Permissions → Add → User Permission**.

| Field | Value |
|-------|-------|
| Path | `/` |
| User | `iac-provisioner@pve` |
| Role | `IaCProvisioner` |
| Propagate | Checked |

2. Click **Add**.

#### Generate an API Token (Recommended for OpenTofu)

Using an API token instead of a password is more secure for automation because tokens can be
independently revoked without changing the account password.

1. Navigate to **Datacenter → Permissions → API Tokens → Add**.

| Field | Value |
|-------|-------|
| User | `iac-provisioner@pve` |
| Token ID | `opentofu` |
| Privilege Separation | Unchecked (token inherits user's permissions) |
| Comment | `Token used by OpenTofu for resource provisioning` |

2. Click **Add**. **Copy the displayed token secret immediately** — it is shown only once.

Store the token in a secrets manager or a `.tfvars` file that is excluded from version control
(`.gitignore`).

#### Configure OpenTofu to Use the Token

In your OpenTofu configuration, reference the token via variables:

```hcl
provider "proxmox" {
  endpoint  = "https://<node-ip>:8006"
  api_token = "iac-provisioner@pve!opentofu=<token-secret>"
  insecure  = true   # set to false once TLS is properly configured
}
```

## Summary

| Step | What Was Done |
|------|---------------|
| Hardware | Two physical machines (mini PC + laptop) connected via Ethernet |
| Proxmox Install | Proxmox VE installed independently on each machine with static IPs |
| Clustering | Both nodes joined into a single cluster named `homelab` |
| Upgrade & Nag | ProxMenux used to upgrade Proxmox VE and remove the license pop-up on each node |

for Kubernetes cluster deployment using OpenTofu and Talos Linux.
